# 产品与技术架构

## 一、产品目标

这个系统的核心不是“帮你记会议”，而是“把高频融资沟通变成组织资产”。

围绕你的场景，系统需要形成一个闭环：

1. 记录真实对话
2. 找出投资人反复问的问题
3. 分析你的回答是否完整和一致
4. 提炼出高质量、可复制的话术
5. 把这些话术持续用于新人培训
6. 新的对话再反哺知识库

## 二、推荐用户角色

- `创始人/核心发言人`
  录音、查看复盘、查看自身回答风格变化。
- `投融资负责人`
  查看高频问题、统一口径、维护标准答案版本。
- `培训对象/新人`
  学习最新版本的话术，基于历史真实问题做演练。
- `管理员`
  配置模型、数据保留策略、标签体系和权限。

## 三、核心使用流程

### 1. 会前

- 选择会议类型：`路演 / 一对一 / 尽调 / 跟进`
- 可选填写公司、基金、投资阶段、会议目标

### 2. 会中

- 手机或电脑端开始录音
- 本地缓存音频
- 会议结束后自动上传

### 3. 会后自动处理

- 语音转写
- 说话人分离
- 对话切段
- 识别投资人问题
- 识别你的回答
- 将问题聚类归档到问题库
- 评估本次回答质量
- 生成会议复盘与改进建议

### 4. 长期沉淀

- 累计高频问题排名
- 形成标准答案版本库
- 识别你长期的回答风格
- 自动生成新人培训话术与模拟问答材料

## 四、为什么推荐 Flutter

你的要求是“手机和电脑设备都能独立运行”。如果优先级是尽快把产品做起来，而不是极致平台定制，`Flutter` 是最合适的第一选择：

- 一套 UI 代码覆盖移动端和桌面端
- 录音、文件上传、权限申请、列表页、复盘页都容易统一开发
- 后续如果要加平板端，也可以直接复用
- 开发速度快，适合验证产品闭环

如果未来对桌面端原生能力、后台常驻录音或复杂系统集成要求更高，可以再把桌面端单独拆成 `Tauri` 或原生客户端。

## 六、服务拆分建议

### 1. Client App

负责：

- 录音
- 上传
- 会议列表
- 转写结果查看
- 复盘建议查看
- 培训话术阅读

### 2. API 服务

负责：

- 登录鉴权
- 会议管理
- 文件签名上传
- 查询复盘结果
- 查询问题库与培训内容

### 3. AI 处理服务

负责：

- 音频转写
- 说话人分离
- 问题识别与回答配对
- 回答打分
- 风格总结
- 标准话术生成

### 4. Knowledge 服务

负责：

- 历史问题聚类
- 标准答案版本化
- 训练材料生成
- 相似问题检索

## 七、AI 能力拆解

### 1. 语音转写

输入：音频文件  
输出：按时间轴切分的文本段落

关键要求：

- 支持中英文混说
- 支持融资场景专有词
- 尽量区分多位说话人

### 2. 问题抽取

输入：转写文本  
输出：结构化问题记录

目标不是简单抽“疑问句”，而是识别真正的投资人关注点，例如：

- 商业模式
- 增长数据
- CAC / LTV
- 壁垒
- 团队背景
- 竞争格局
- 融资用途
- 财务预测
- 合规风险

### 3. 回答评估

每个回答建议至少从以下维度打分：

- `完整性`
- `清晰度`
- `一致性`
- `证据充分度`
- `风险表达是否得当`
- `是否偏题`
- `是否过长`

### 4. 风格画像

系统长期总结你的回答风格，例如：

- 偏数据驱动还是偏故事驱动
- 偏谨慎还是偏激进
- 偏简洁还是偏发散
- 是否容易在某些主题下回答不足

### 5. 培训话术生成

输出不是一段死板标准答案，而应包括：

- 核心表达框架
- 推荐关键词
- 不该说的点
- 可展开的数据点
- 遇到追问时的分层回答方式

## 八、建议的数据处理策略

### 1. 原始数据保留

保留：

- 原始音频
- 原始转写
- AI 结构化结果
- 人工修订版本

原因：

- 后续模型升级后可以重新跑分析
- 可以回溯某条建议是如何产生的

### 2. 人工可修订

AI 的抽取和评估一定要允许人工修正：

- 某句话是不是投资人问题
- 某个回答是否归到错误主题
- 标准话术是否要升级为正式版本

这一步很重要，因为你最终需要的是“组织可控的话术资产”，而不是黑盒结论。

## 九、MVP 功能清单

### 必做

- 录音与上传
- 会议列表
- 转写结果展示
- 问答抽取
- 问题库
- 单次会议回答复盘

### 第二阶段

- 高相似问题自动归并
- 新人培训模式
- 标准答案版本审批

### 第三阶段

- 团队成员回答风格对比
- CRM / 日历 / 邮件系统联动

## 十、关键风险与应对

### 1. 转写质量不稳定

应对：

- 引入术语词典
- 保存原文与修订文双版本
- 支持人工校对重点段落

### 2. 问答配对错误

应对：

- 先使用规则 + 模型双重校验
- 在复盘页支持快速纠正

### 3. 生成的话术越来越空泛

应对：

- 训练材料必须引用真实历史回答片段和结构化结论
- 强制保留证据来源和版本号

### 4. 数据合规与隐私

应对：

- 录音前合规提醒
- 数据脱敏
- 音频加密存储
- 可配置保留期限

## 十一、建议的技术栈

- 客户端：`Flutter`
- 后端 API：`FastAPI`
- 异步任务：`Celery / Dramatiq / Temporal`
- 数据库：`PostgreSQL`
- 向量检索：`pgvector`
- 缓存与任务队列：`Redis`
- 对象存储：`S3 兼容存储`
- 鉴权：`JWT + Refresh Token`
- 模型接入：统一 `LLM Gateway`

## 十二、我建议你先做的最小闭环

只验证一个问题：

“系统能不能从一次真实融资沟通中，稳定产出有价值的复盘建议和可复用话术？”

为此，第一版只需跑通：

1. 上传一段录音
2. 转写成文本
3. 识别 5 到 20 个问答对
4. 给出每个回答的不足和改进建议
5. 生成一份本场会议的统一复盘
6. 把高频问题归入知识库
7. 产出一版培训话术草稿

只要这个链路成立，后面的跨端体验、权限、报表都可以逐步补齐。
